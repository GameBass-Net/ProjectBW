# tech — 절차적 월드 생성 설계

> 본체 문서. 메커니즘 중심. 컨텐츠 관점은 [../03-content.md](../03-content.md), NFR은 [../02-requirements.md](../02-requirements.md).

## D1. 생성 시점 — 런타임 유한 스트리밍 (확정)
- 월드맵 데이터 없으면 **러프 월드맵**을 절차적으로 먼저 생성.
- 맵을 일정 간격으로 쪼개 **Zone**으로 구분.
- 각 존 디테일 시드 = `worldSeed + Zone좌표` 조합 → 존별 디테일 절차 생성. (단, 베이스 높이는 전역좌표 노이즈 — C3 참조)
- 존 데이터 있으면 로드, 없으면 생성(load-or-generate).
- 레퍼런스: Valheim(유한 월드 + 존 스트리밍).
- **존 그리드**: 월드는 균일 그리드(예: 월드 16384m / 존 128m → 128×128개 존, 각 128m²). 16384는 예시, 최종 크기 추후.
- **구조 규율**: "특정 섬/존"을 코드에 하드코딩하지 말 것. **전체 존 그리드를 일반적으로** 다루고, "어느 존이 육지냐"는 **마스크 데이터**로. → 규모 확장이 데이터 변경만으로 됨.

## 구현 규율 (생성 코어)
- **난수는 MT(Mersenne Twister, 직접 구현한 결정적 RNG)만.** `UnityEngine.Random`/시간 기반 금지.
- 베이스 높이·연속 디테일·바이옴 필드는 **전역 월드좌표**로 평가(C3). 시드는 `worldSeed` 단일, 존 시드는 이산 콘텐츠 전용.
- 생성 코어 = 엔진 비종속 순수 C#(D2/D8), 출력 = 정규 데이터(하이트필드+바이옴 가중치+동굴 SDF)+깔끔한 API.

## 구현 구조 (모듈)
**코어는 Unity 밖 독립 .NET 솔루션.** Unity는 빌드된 DLL만 참조.

- **`Bass.Core`** → `ProjectBW/Core/Bass.Core/` (솔루션 `GameBassLib.sln`). 순수 C#, **UnityEngine·Unity.Mathematics 의존 0.**
  - 멀티타깃 **`netstandard2.1;net10.0`** (Unity=ns2.1, 서버=net10). ImplicitUsings 비활성, Nullable enable.
  - `Bass.Core.MersenneTwister`(std::mt19937 호환), 전역 `FastNoiseLite`(벤더링), `Bass.Core.WorldGeneration.*`(필드/`BiomeAt`/`CaveField`/`WorldGenerator`/Config/출력 구조).
  - 벡터/수학 필요 시 `System.Numerics` 또는 자체 구현(Unity.Mathematics 미사용 — 포팅성). Burst 미사용(결정성).
- **테스트** → `Bass.Core.Test`(xUnit, net10). `dotnet test`로 검증, Unity 밖.
- **DLL 전달**: Release/ns2.1 산출물을 **수동으로** `Client/Assets/Plugins/Bass.Core.dll`에 복사·커밋(빌드 파이프라인 X). Unity는 이 DLL만 사용.
  - 예: `dotnet build Core/Bass.Core -c Release -f netstandard2.1` → `bin/Release/netstandard2.1/Bass.Core.dll` → `Client/Assets/Plugins/`.
- **Unity 통합 레이어**(M2) → `Client/Assets/_ProjectBW/Scripts/...`. `Bass.Core.dll`(Plugins) + Digger 참조. `ZoneBuilder`(코어 질의→Terrain/splat/Digger), `WorldManager`(존 그리드+육지 마스크, P1 정적 로드). (asmdef/ns명 M2에서 확정)

## D2. 생성 엔진 — 커스텀 순수 C# 코어 (확정)
- **생성 코어 = 엔진 비종속 순수 C# 직접 구현.** Unity 종속 생성 에셋 미사용.
- 이유: 서버 포팅 가드레일(순수 C#)과 일관 + 생성 방식 이해/통제 + MT 직접 구현과 결.
- 에셋/외부 자료는 **아이디어 소스로만** 참조. 노이즈: FastNoiseLite(MIT) 등 후보.
- 존/스트리밍/세이브 매니저도 커스텀.

## D5. 월드 생성 파이프라인 (확정 — C1~C4 조건부)
1. **러프 월드 하이트맵** 생성(저해상, 대지형 골격).
2. **바이옴 맵** 생성(멀티노이즈 온도/습도 + 러프 고도 + 중심거리 가중).
3. **바이옴 맵에 의한 러프 하이트맵 보정**(산악↑, 해양/늪↓).
4. → 확정 하이트맵 기반으로 **존별 디테일** 생성.

**조건(합의 완료):**
- **C1. 고도↔바이옴 피드백**: 바이옴 분류는 **보정 전 러프 고도**로, 보정은 **바이옴과 모순 없는 범위**로 제한(필요 시 1회 반복 수렴).
- **C2. 연속 블렌딩 입력은 연속 필드로**: 양자화된 바이옴 인덱스 맵을 업샘플하면 경계 뭉갬. 결정 입력(온도/습도/고도)을 **연속 노이즈 필드로 보관**, 디테일 단계에서 점마다 재평가→분류→블렌딩.
- **C3. 이음매 없는 경계 = 전역좌표 노이즈**: 베이스+연속 디테일은 **전역 월드좌표**에서 평가(`worldSeed`가 전역 필드 결정). 존은 그 전역 함수의 "창". **"존 시드(worldSeed+존좌표)"는 경계 연속성 불필요한 존 로컬 이산 콘텐츠(동굴 레이아웃, POI/프롭)에만.**
- **C4. 제약형 보정(클램프)의 바이옴 경계 절벽**: 클램프는 점별 함수라 **존 이음매는 안전**. 단 바이옴 경계에 절벽 생길 수 있음(타일링 무관, 지형 품질). 완화: ① 하드클램프 대신 연속 가중치 블렌딩, ② 제약을 **생성 단계에 흡수**(해변은 애초에 낮게). 하드 제약이 디테일 후에도 유지돼야 하면 **디테일 후 재클램프 또는 해안 디테일 진폭 축소.**

> 보류: 바이옴 최소 크기/스케일 — 추후(만들어보고 조정).

## D4. 바이옴 — 결정/단위/구동 (확정)
- **적용 단위: 연속 바이옴(A-2)** — 존 내부에서도 변함, 경계·내부 블렌딩.
- **결정 = 위치 질의 가능한 순수 함수** `BiomeAt(worldPos)`: 전역 필드를 그 위치에서 평가→분류/블렌딩. `worldSeed`+좌표만으로 결정 → 플레이어 위치·스폰·게임플레이 어디서든 질의. 별도 저장 불필요(러프 맵은 원경/캐시용만, D7).
- **계산 위치**: 러프 월드맵 단계에서 전역으로 계산 → 존 상속.
- **구동 대상(확정)**: ① 지표/동굴벽 텍스처, ② 하이트맵 셰이핑, ③ 동굴 스타일/밀도. (④~⑦ 추후)

#### 맵 레이어 모델 (전부 전역 연속 필드, worldPos 샘플)
| 레이어 | 역할 |
|---|---|
| **Continentalness(대륙도)** | 바다/해안/내륙, 섬 형태 |
| **Elevation(고도)** | 러프 하이트맵(대륙도 입력) |
| **Temperature(온도)** | 기후(중심거리/위도 영향) |
| **Moisture(습도)** | 건/습 |
| **Erosion/험준도** | 평탄 vs 험준(고도와 별개, 산악·암석지대 구분). **P1은 고도로 퉁치고 후일 분리.** |
| → **Biome(파생)** | 아래 2단계 분류 |

#### 바이옴 판정 (2단계)
1. **대륙도 → 바다 / 육지** (해안은 별도 바이옴 X, 육지/바다 전이대로 텍스처·저고도 처리).
2. **육지면** → 고도+온도+습도(+Erosion)+중심거리 → 세부 바이옴.

- **바이옴 종(5)**: 초원 / 설산 / 사막 / 암석지대 + **Ocean**(대양). → [../03-content.md](../03-content.md).
- **P1 섬 형태**: 대륙도 노이즈 대신 **명시적 섬 마스크(반경 감쇠)** 사용(작고 통제 쉬움). 전체 월드는 대륙도 노이즈.

## D3. 동굴(절차적) — 카빙/규모/분포 (확정)
- **카빙**: 존 첫 생성 시 절차적 동굴 경로(3D 노이즈/펄린 웜)를 깎고 **존 데이터에 영속화** → 이후 로드만.
- **규모**: 도적/몬스터 소굴 수준 소~중. 대규모 지하 범위 외.
- **분포**: 전 존, 바이옴별 스타일/밀도 분기.
- **기술 제약**: Digger 동굴은 터레인 솔리드(`y=0~terrainHeight`) 안에서만 → 동굴 깊이는 지표 고도 예산에 묶임.
- **경계 무결(중요)**: 절차적 동굴은 "존 소유 객체"가 아니라 **전역 동굴 필드(월드좌표 SDF)**로 정의하고 각 존이 자기 볼륨에서 평가 → 경계 자동 일치. 구현 훅 = Digger **`IVoxelGenerator`**(복셀 채울 때 전역 SDF 평가). → 프로토타입 단일 존은 단순 Modify로 시작, 스트리밍 단계에서 전역 필드로 전환.
- **런타임 유저 디깅(경계 가로지름)**: `DiggerMasterRuntime.Modify`가 `diggerSystems` 전체를 순회 → 브러시가 겹치는 양쪽 터레인에 동시 적용(복셀 월드공간 정렬로 이음매 없음). 단 **두 존 모두 로드** 시. 영향받은 두 존 데이터 모두 영속화. 이웃 미로드 시 그쪽 면 누락 → 스트리밍 반경으로 이웃 항상 로드 보장(폴백: 미로드행 편집 기록 후 로드 시 적용).

## D7. 영속성 / 세이브 (확정)
- **방식: 시드 재생성 + 델타만 저장.** 베이스 지형·절차적 동굴은 시드+전역좌표로 결정적 재생성 → 저장 안 함.
- **저장 대상**: 플레이어 변경분만(곡괭이질=Digger 수정, 설치 구조물, 엔티티 상태).
- **저장 범위**: 플레이어 존+인접 존만. 미방문/미확인 존은 저장 대상 아님.
- **예외**: **러프 하이트맵은 저장**(저해상·가벼움, 원경 LOD 배경 재사용).
- **전제**: 재생성 결정성 + **생성 버전 스탬프**(알고리즘 변경 시 구세이브 무효 판정).

## 텍스처 레이어 (프로토타입, 7장 — URP 8장 내, MicroSplat 불필요)
경사/고도 규칙으로 바위·동굴벽을 **공유**해 레이어 폭발 방지.

| # | 레이어 | 용도 | 공유 |
|---|---|---|---|
| 1 | Grass | 초원 바닥 | 초원 전용 |
| 2 | Sand | 사막 바닥 | 사막 전용 |
| 3 | Snow | 설산 바닥 | 설산 전용 |
| 4 | Gravel | 암석지대 바닥 | 암석지대 주력 |
| 5 | Cliff/Rock | 급경사=전 바이옴 공통 + 설산 노출암 + 암석지대 | 공유 |
| 6 | Dirt | 전이/저지대(바이옴 경계) | 공유 |
| 7 | Cave wall | Digger 동굴 내벽(전 바이옴) | 공유 |

**블렌딩 규칙**: 바닥레이어=바이옴 가중치 / 바위(5)=경사도 오버레이 / 흙(6)=경계·저지대 전이 / 동굴벽(7)=Digger texture index.
**Ocean은 추가 레이어 불필요**: 물은 별도 평면(Stylized Water 2), 해저는 모래/자갈(1·4) 재사용. → 7장 유지.
**확장 시**: 숲/늪 추가나 바이옴별 바위 변형 원하면 8장 초과 → MicroSplat 도입.
