# tech — 데디케이티드 서버 / 네트워크

> **프로토타입 범위 밖.** 추후 결정. 지금은 방향성과 가드레일만.

## 멀티플레이 범위 (확정)
- **싱글 + 소규모 코옵. 최대 동접 ~16명.** 지속 월드 데디 서버. **MMO 아님.**
- **MMO 제외 근거**: 네트워크 자체는 가능하나 **Digger식 복셀 지형을 포기해야 함**(대규모 동접에서 복셀 지형 동기화·충돌 비현실적). Digger 지형을 유지하려면 코옵 규모가 맞음.
- ~16 동접은 **Unity 헤드리스(A안)로 충분**. 순수 C# 코어(C안)는 필수가 아닌 최적화 선택지.

## 현 결정 (D8)
- **프로토타입은 데디 서버 고려 안 함. 싱글 클라(Unity)만.** 서버·네트코드·자체 충돌은 프로토타입 이후.
- 프로토타입에선 **Digger 그대로 클라에서 사용**(메싱+콜라이더+Unity 물리) → 최소 공수.
- **가드레일(지금 지킴)**: 월드 생성 로직을 **엔진 비종속 순수 C#**으로 격리(`UnityEngine` 의존 0, MT). 후일 서버/포팅 길 보존.
- **결정성**: 메르센 트위스터 C# 직접 구현 완료 → 서버/클라 동일 월드 합의의 RNG 퍼즐 해결.

## 핵심 긴장
지금 스택(Digger·Unity Terrain·Unity 물리·SW2)은 **Unity 종속**.
- 월드 생성 로직(노이즈/바이옴)은 순수 수학 → **포팅 가능**.
- 그러나 **Digger 복셀 충돌·지형 충돌·캐릭터 물리·물**은 Unity 없이 권위 시뮬 불가 → 비-Unity 순수 서버는 충돌 재구현(대공사).

## 서버 아키텍처 2안 (프로토타입 이후 택1)
- **(A) Unity 헤드리스 + Digger** — 단순, 스택 재사용, 서버 권위. 단 Unity 루프 오버헤드 + Digger 서버구동 필요.
- **(C) 순수 C# 코어 + 자체 복셀/SDF 충돌, Digger 클라 전용** — 경량/포팅 용이, 사용자 최적화 선호와 일치. 단 **자체 충돌 레이어 구현이 숙제**(서버는 Digger 미사용, 클라만 렌더). 마크 계열 서버가 풀 물리엔진 없이 복셀-AABB로 충돌하는 방식.
- C++ 포팅: 가능하면 좋으나 현실적 난도 높음(부동소수 결정성), C#도 수용.
